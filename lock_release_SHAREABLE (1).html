01
Monitor log file continuously
qad_lock_mon.log  ·  /tmp/locking/  ·  all shifts
▼

The locking alerts are received in /tmp/locking/qad_lock_mon.log. This file is monitored continuously across all shifts for any signs of locking alerts.

/tmp/locking/qad_lock_mon.log
📁No action required until a lock alert appears. Monitoring is passive until an alert is recorded in the file.
02
Lock alert detected in log
Record updated — Held time, Usr, PID, Table, Blocker info
▼

When a lock occurs, the lock monitor script updates the log file. Key fields in each alert:

Held (seconds) Usr / PID (waiter) Trans ID Table / RECID Blocking Usr / PID (blocker)
Sample log entry
# Waiter waiting, Blocker holding the lock Held > :0 sec, Usr 308 PID 22573 Trans 1743690315 Table code_mstr RECID 1307660250 (ABL) Blocking Usr 245 PID 1658 (ABL) Held > :0 sec, Usr 308 PID 22573 Trans 1743789023 Table code_mstr RECID 1307660250 (ABL) Blocking Usr 378 PID 30298 (ABL)
⚠Blocker PID is the session holding the lock — this is the candidate for disconnection. Waiter PID is the session waiting to get access.
03
180-second threshold decision
Auto-trigger at 180s  ·  OR explicit Medline request
▼
LOCK DURATION THRESHOLD INDICATOR
0s
180s → ACT
< 180s: Continue monitoring  |  ≥ 180s: Begin disconnect process
▲ THRESHOLD CROSSED (≥180s)
Lock monitor script updates file with protrace details of both PIDs. Proceed immediately to Step 04. Do not delay.
👤 EXPLICIT MEDLINE REQUEST
If Medline personnel explicitly request disconnection before 180s, proceed to Step 04 immediately without waiting.
✗ NOT YET CROSSED
Lock is under 180s, no Medline request. Continue monitoring the log. Return to Step 02.
04
Check ABL Stack Trace (protrace)
Identify program executed by blocker & waiter PIDs
▼

Once threshold is crossed, the lock monitor script adds protrace details to the log. Check the ABL Stack Trace section to identify the program running under both PIDs.

🔎The line marked with --> is the currently executing program. Note the program names for both the blocker and waiter before proceeding to Step 05.
Sample ABL Stack Trace
** ABL Stack Trace ** --> us/av/avmkdrp7.r at line 760 (/extl/prod/2017ee/dist/mfg-customizations/us/av/avmkdrp7.r) us/av/avwinrun.p at line 3704 (/extl/prod/2017ee/dist/mfg-customizations/us/av/avwinrun.r) us/av/avtrig.p at line 4317 (/extl/prod/2017ee/dist/mfg-customizations/us/av/avtrig.r) us/gp/gpwinrun.p at line 558 (/extl/prod/2017ee/dist/mfg-customizations/us/gp/gpwinrun.r) mfwb0la.p at line 9885 (/extl/prod/2017ee/dist/mfg/mfwb0la.r) com/qad/qra/core/ClientBootstrap.p at line 61
05
Check program name & user — MANDATORY
Run ABL command before disconnecting any PID
▼
🚨This step is mandatory and must never be skipped. Always verify the program name and user ID before disconnecting. Run the command for both the blocker and waiter PIDs.

Replace 8314 with the actual PID number of the blocker or waiter:

ABL Command — Check Program Name & User
/* Replace 8314 with the actual PID number */ FIND FIRST _connect where _Connect-Pid = 8314 no-lock no-error. if available _connect then do: find last mon_mstr where mon__qadi01 = _Connect-Usr no-lock no-error. message mon_userid mon_program . end.
📋mon_userid — the logged-in user of this session  |  mon_program — the program currently running. Cross-reference with the ABL stack trace from Step 04.
06
Grep process details of PIDs
ps -ef | grep <PID>  —  run for both blocker & waiter
▼

Use the following OS command to identify the process type of both the blocker and the waiter PIDs. This is the key input for determining which case applies in Step 07.

# Run for BOTH blocker and waiter PIDs ps -ef | grep -i <PID> # Example: ps -ef | grep -i 22573

Look for the process type in the output:

uiprod mfgbatch appserver agent as-mrp ★
⚠as-mrp is a special case (Case 6) — if both PIDs are as-mrp, contact the Medline DBA before taking any action.
07
Determine the case & take action
Match blocker & waiter process types to one of 6 cases
▼

Match the blocker and waiter process types to the correct case below and follow the prescribed action:

CASE 1
Both sessions are uiprod
Blocker = uiprod   |   Waiter = uiprod
→ Disconnect the blocker PID directly
CASE 2
Blocker is uiprod — Waiter is mfgbatch
Blocker = uiprod   |   Waiter = mfgbatch
→ Disconnect the blocker PID
CASE 3
Both sessions are mfgbatch
Blocker = mfgbatch   |   Waiter = mfgbatch
Check which batch job runs most frequently.
→ Disconnect the more frequent one
CASE 4
Blocker is uiprod — Waiter is appserver agent
Blocker = uiprod   |   Waiter = appserver agent
→ Disconnect the blocker PID
CASE 5
Both sessions are appserver agents
Blocker = appserver agent   |   Waiter = appserver agent
Check log files of the appserver instances to identify process type:
If MSW / PSW process
→ Inform Medline DBA immediately. Do not disconnect without DBA authorization.
If Browse process
→ Inform the end user. Get their approval, then proceed with disconnection.
CASE 6 ★
Both sessions are as-mrp — SPECIAL CASE
Blocker = as-mrp   |   Waiter = as-mrp    (MRP process — critical)
🚨Mandatory: Contact Medline DBA before any action. MRP is a critical planning process. Do not disconnect without explicit DBA confirmation.
08
Pre-disconnect checklist & execute disconnect
Complete all items before disconnecting the PID
▼

Confirm every item below before executing the disconnection. Click each item to mark it complete.

✓
ABL Stack Trace reviewed for both blocker and waiter PIDs (Step 04)
✓
Program name and user verified using the ABL command for both PIDs (Step 05)
✓
PID process type confirmed via ps -ef | grep command (Step 06)
✓
Correct case identified and matched to action (Step 07)
✓

✓
If browse process (Case 5): End user informed and approval received
✅Once all items are confirmed, proceed with disconnecting the identified PID as per the case. Confirm the lock is released after disconnection.
09
Lock released — resume monitoring
Confirm resolution · return to continuous log monitoring
▼

After the PID is disconnected, confirm the lock is no longer in the log, then resume monitoring /tmp/locking/qad_lock_mon.log across all shifts for the next alert.
